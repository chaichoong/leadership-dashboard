🏗 1. System Architecture

🌐 Component Overview

The Leadership Dashboard is a client-side HTML/JavaScript application that combines financial data from Airtable with invoice data from Gmail. It runs entirely in the browser with no back-end server, connecting directly to external APIs via personal access tokens and a Google Apps Script web app.

The dashboard reads from six Airtable tables: Transactions, Tenancies, Tenants, Rental Units, Fixed Costs, and Tasks. It also fetches invoice data from a deployed Google Apps Script that reads emails labelled 3. to pay.

🔑 2. Access & Authentication

🚪 Opening the Dashboard
  1. Open the dashboard.html file from Google Drive (double-click or open in browser).
  2. On first load, the login screen appears requesting your Airtable Personal Access Token (PAT).
  3. Enter your PAT and click Connect. The dashboard loads all data from Airtable and Gmail.
  4. The token is stored in sessionStorage and persists until you close the browser tab.
  5. If the session expires or you close the tab, re-enter the PAT on next visit.
🔒
Session-only storage: Your PAT is stored in sessionStorage only. It is automatically cleared when the browser tab is closed. No credentials are persisted to disk or sent to any third-party service.
⚠️
Security note: No credentials are sent anywhere except directly to the Airtable API (api.airtable.com) and the Google Apps Script web app URL. There is no intermediary server.

🏠 5. Portfolio & Tenancy Metrics

🏘 Portfolio Metrics

Portfolio metrics are drawn from the Rental Units table and show property occupancy at a glance:

  • Total Units: Count of all rental units in the portfolio
  • Occupied: Units with an active tenancy assigned
  • Void: Units currently without a tenant
  • Occupancy Rate: Percentage of occupied vs total units
👥 Tenancy Status Breakdown

Tenancy metrics show the payment health of active tenants from the Tenancies and Tenants tables:

  • In Payment: Tenants currently paying rent on schedule
  • CFV (Consideration for Vulnerability): Tenants flagged for review
  • CFV Actioned: CFV cases where action has been taken
🚦 Traffic Light System

Key metrics are displayed using a traffic light system to provide at-a-glance health indicators. Each metric is shown as a coloured chip:

Green — On target Amber — Needs attention Red — Action required

Thresholds are applied to occupancy rate, CFV count, void count, and cash flow closing balance. Green indicates healthy operation, amber signals early warning, and red requires immediate review.

📅 6. 31-Day Cash Flow Forecast

📈 Forecast Logic

The cash flow forecast projects daily inflows and outflows for the next 31 days, starting from today’s combined opening balance. It draws on tenancy rent schedules, fixed cost due dates, and reconciliation data to avoid double-counting.

A

Income Projection

Inflows

How rental income is projected

  1. All active tenancies are retrieved from the Tenancies table with their rent amount and payment day.
  2. Each tenant’s status is checked — only tenants marked as “In Payment” contribute to the income forecast.
  3. Tenants receiving Universal Credit (UC) are marked separately; their payment may arrive on different dates to direct-pay tenants.
  4. For each day in the 31-day window, the system calculates whether a rent payment falls on that day based on the tenant’s payment frequency and due date.
  5. The projected amount is checked against reconciled transactions to avoid counting payments that have already arrived.
B

Cost Projection

Outflows

How fixed costs are projected

  1. All active fixed costs are loaded from the Fixed Costs table, including due day and payment frequency (weekly, monthly, quarterly, annually).
  2. For each cost, the next due date is calculated based on its frequency and last known payment date.
  3. Costs are placed on the appropriate day within the 31-day forecast window.
  4. Like income, projected costs are cross-referenced against reconciled transactions to prevent double-counting.

🔍 7. Reconciliation Matching Logic

🚨
Critical system: The reconciliation matching logic is essential for preventing double-counting of transactions that have already been paid. If a projected income or cost has already been settled in the bank, the matching engine ensures it is excluded from the forecast to maintain accuracy.
📆 Date Matching Window
Parameter Value Purpose
Lookback 5 days before expected date Catches early payments (e.g., rent paid a few days ahead)
Look-ahead 5 days after expected date Catches late payments (e.g., delayed bank transfers)
↑

Inflow Matching (3-Tier)

Income

Tier 1: Strong Match

Two or more keywords from the tenant name or reference appear in the bank transaction description. This is the highest-confidence match.

Tier 2: Good Match

One keyword from the tenant name plus the amount matches within 30% of the expected rent. Provides reasonable confidence with a partial name match.

Tier 3: Fallback Match

The transaction amount matches the expected rent exactly (to the penny). Used only when no keyword matches are found, as an amount-only match is the weakest signal.

↓

Outflow Matching

Costs

Matching Criteria

Outflow matching requires all three conditions to be met:

  • Keyword match: At least one keyword from the cost name appears in the transaction description
  • Amount match: The transaction amount is within 33% of the expected cost, or within £1 (whichever is more generous)
  • Sub-category compatibility: The transaction’s Airtable sub-category is compatible with the fixed cost’s category
💡
Why sub-category matters: The sub-category check prevents false matches where two different costs share a similar name or amount. For example, two utility bills of similar value would not be confused if their sub-categories differ.

🔧 8. Balance Calculator

🧮 Interactive What-If Tool

The Balance Calculator is an interactive what-if tool that allows you to model different cash flow scenarios by selectively including or excluding individual inflows and outflows from the 31-day forecast. It is accessed by clicking the wrench icon on the dashboard toolbar.

How It Works

  1. Click the wrench icon to open the Balance Calculator panel.
  2. Each projected inflow and outflow is listed with a checkbox. Ticked items are included in the forecast; unticked items are excluded.
  3. Select or deselect individual items to see how the closing balance changes in real time.
  4. Use the filter buttons to quickly adjust selections: Select All, Deselect All, Inflows Only, or Outflows Only.
  5. The selected state is saved to sessionStorage so it persists during your browser session.
  6. Selections auto-reset daily to ensure each day starts with a clean forecast.
💾
State persistence: Your Balance Calculator selections are stored in sessionStorage and will persist while your browser tab is open. They automatically reset at midnight to provide a fresh forecast each day.

📅 Credit Card Payment Logic

The dashboard calculates recommended credit card payments based on several business rules:

  • Friday-based payments: Payments are scheduled to coincide with Fridays to align with business cash flow cycles
  • 7-day lookahead: The system looks 7 days ahead to identify upcoming payment due dates
  • £750 buffer: A buffer of £750 is maintained in the business account before credit card payments are recommended
  • Minimum payment: Either 2% of the outstanding balance or £25, whichever is greater

Defers while active: The smart refresh system monitors user activity (clicks, keystrokes, and input). ⚙️ 12. Google Apps Script Setup

🚀 Deployment Steps
  1. Open Google Apps Script at script.google.com and create a new project (or open the existing one).
  2. Paste the Apps Script code that handles reading emails from the 3. to pay label, extracting invoice fields, and processing markPaid actions.
  3. Click Deploy > New deployment and select Web app as the deployment type.
  4. Set Execute as: Me (Kevin’s Google account) and Who has access: Anyone (so the dashboard can call it without OAuth).
  5. Copy the Web app URL and paste it into the dashboard’s GMAIL_SCRIPT_URL configuration constant.
  6. Test by loading the Invoices tab — invoices should appear with a Live badge.
📧 What the Script Does
  • Reads emails: Fetches all threads with the 3. to pay Gmail label
  • Extracts fields: Parses the email subject, sender, body, and attachments to extract payee, amount, due date, and reference
  • OCR for PDFs: If a PDF attachment is present, the script can use Google Drive’s built-in OCR to extract text from the PDF for field extraction
  • markPaid action: When called with a markPaid action, the script moves the email thread from 3. to pay to 4: paid

🛠️ 13. Troubleshooting

🔧 General Issues
Dashboard shows login screen / “Session expired”
Your Airtable PAT has expired or the browser tab was closed. Re-enter your PAT on the login screen. The token is stored in sessionStorage and does not persist after the tab is closed.
Refresh interrupts my work / reloads while editing
The smart refresh system should defer while you are active. If it is not detecting your activity, ensure you have interacted with the dashboard (click, keystroke, or input) within the last 15 minutes. The idle threshold must be exceeded before an auto-refresh fires.
A reconciled payment still appears in the forecast
The reconciliation matching engine uses a ±5 day window. If the payment fell outside this window (e.g., more than 5 days early or late), it will not be matched. Check the transaction date in Airtable and adjust if needed.
