## Notes for tuning the sweep (self-maintaining observations)

1. **`compliance` and `how-it-works` are `standalone` registry entries, not sidebar tabs.** They have no `tab-<id>` panel in index.html and `switchTab` is never wired to them (no sidebar `onclick`). The naive `tab-<id>` check flags them as "no panel" — but that is by design; they are full pages reached by direct link (e.g. from the sitemap). Both were verified by loading their standalone HTML directly and both render clean. Future sweeps: for a registry entry whose `standalone` value is a distinct `.html` file (not an `index.html#hash` and not wired into a `tab-` panel), test the standalone URL directly instead of `switchTab`.

2. **Hidden-tab `requestAnimationFrame` pause.** The automated preview tab runs with `document.visibilityState === 'hidden'`, so `requestAnimationFrame` callbacks never fire. `renderCFVTab()` (js/cfv.js:383-384) awaits two rAF frames to paint its spinner before the heavy `detectCFVs()` sweep; with rAF paused, that await never resolves and the CFV panel appears frozen on "LOADING CASH FLOW VOIDS…". This is a **test-harness artifact, not a production bug** — a real user's foreground tab fires rAF at ~60fps and CFV renders instantly. Confirmed healthy by calling `detectCFVs()` directly (6 CFVs, real data, no error). Any future tab that gates render behind rAF will show the same false "stuck" symptom in this sweep; verify by running the tab's core render function directly before flagging.

3. **Internal-health dots below full on 5 tabs** (wealth 5/7, income 5/6, costs 3/5, sitemap 3/7, skills 4/5). These are each page's own self-reported data-sync / automation checks, not JS errors — every one of those tabs rendered real content with zero console errors. Not classified as failures. Worth a glance if these ratios keep dropping, as it can indicate a stale nightly automation or a data feed lagging.
