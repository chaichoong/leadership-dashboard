### cfv — PASS (logic verified; UI stall is a sweep-harness artifact, NOT a production fault)
- The CFV panel stayed on its "LOADING CASH FLOW VOIDS…" placeholder in this sweep and never filled in.
- Root cause diagnosed: `renderCFVTab()` awaits two `requestAnimationFrame` callbacks before rendering (js/cfv.js:383-384, intentionally, to paint the spinner before the blocking sweep). In this automated preview the tab runs with `document.visibilityState === "hidden"`, and **requestAnimationFrame never fires when the tab is not painting** (confirmed: an rAF callback did not fire within 2s). So the render is gated forever here.
- Underlying logic is healthy: `detectCFVs()` runs in 21ms and returns 6 CFVs with zero errors. A real user's visible/foreground browser fires rAF in ~16ms and CFV renders normally. No production impact.

## Sweep-tuning notes (the sweep asked me to flag shape changes)
1. **The sweep runs with a hidden, non-painting tab** (`visibilityState = "hidden"`), so any render path gated on `requestAnimationFrame` stalls and looks broken — today that was CFV. Options: force the preview tab visible before walking, or short-circuit the rAF wait when hidden, or special-case CFV to call `detectCFVs()` for verification instead of relying on the painted table. Without this, CFV will "fail to render" every run despite being healthy in production.
2. **`compliance` and `how-it-works` are `standalone` full pages, not `tab-<id>` panels**, and are not in the sidebar. The `tab-<id>` container assumption reports them as MISSING. The sweep should treat registry entries whose `standalone` is a real file (not `index.html#<id>`) and which have no sidebar item as full-page HTTP checks (200 + non-empty + title), which is how I verified them today.
