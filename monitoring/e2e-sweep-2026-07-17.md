## Failures needing attention

### [WARNING] Skills tab — HTTP 422 on every load, silently swallowed

**Symptom:** every load of the Skills tab fires a failed request to Airtable:
`GET .../tblHGNzDmOs59r9QD/recqbcIz2R2griDn3?fields[]=Active%20Skill%20IDs` → **422**.

**Root cause (confirmed by reproducing it against the live API, read-only):** Airtable's
**single-record GET endpoint does not accept the `fields[]` parameter**. The list endpoint
does. Proof:
- single-record GET + `fields[]` → `INVALID_REQUEST_UNKNOWN` (422)
- single-record GET, no params → 200
- **list** endpoint + same `fields[]` → 200

So this is not a bad field name. `'Active Skill IDs'` **does** exist in the schema of
`tblHGNzDmOs59r9QD` (name: "(Deprecated) Settings", type multilineText).

**Where:** `js/skills.js:31` builds the single-record URL with `?fields[]=`. The failure is
then swallowed by `if (!res.ok) return;` at line 33 — no log, no toast.

**Impact:** low, and contained. The tab renders (99 preset + 3 SOP skills load fine) and the
app's own health bar already flags it honestly: *"Active preset settings readable ⚠ — Active
preset record not read — 'Active' filter counts may be stale"*. So the Active/preset filter
counts on that tab can be wrong. No financial data is touched.

**Second, separate issue found while confirming the above:** even with the fetch repaired,
`SETTINGS_RECORD = recqbcIz2R2griDn3` holds an **empty** `Active Skill IDs`. The populated
value lives on a **different** record, `reccOxxLIHkuBodx5` (created 2026-07-15), and its
content is `ec9e382d430c45d1...` — a 64-char hex string, not the JSON array
`JSON.parse` at line 37 expects. So a naive fix to the URL alone would still yield an empty
list (the `catch` at line 37 sets `_activePresetIds = []`). Both need Kevin's call.

Not fixed here — this sweep is read-only and reports rather than repairs.

## False alarms checked and cleared (recorded so future sweeps don't re-chase them)

- **OS iframes "Invalid token" / "Invalid key"**: the Operations and Systemisation iframes
  contain that text in the DOM, but the nodes are `display:none`. Both pages read the same
  `_dlr_pat` key and were live-synced with real data. Not a fault. Reading `innerText` on an

## Tuning suggestions for this sweep's prompt

1. Tab count is now **25**, not 24 — `prospecting` was added. The prompt's parenthetical list
   is stale (the sweep reads the registry live, so coverage was fine; only the comment lags).
2. The prompt says to verify `window.PAT` is a non-empty string. **This is wrong** — `PAT` is
   IIFE-scoped in `js/shared.js`, so `window.PAT` is always `undefined` even when fully
   authenticated. It reads as an auth failure when nothing is wrong. Better signal: the
   `#loadingOverlay` is hidden, `#dashboard` is visible, and `allTransactions.length > 0`.
3. `compliance` and `how-it-works` are `standalone: true` and navigate the window away,
   destroying any in-page walker state. Walk them last or in a separate pass.
