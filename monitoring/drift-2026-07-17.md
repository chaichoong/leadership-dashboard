### STALE_SOP: 1 — `sop.html` §12 "Reconciliation Matching Logic" (NEW, material)

The section describes the cost-matching engine. The code was rewritten; the SOP was not. Almost
every documented rule is now false:

| SOP §12 says | Code actually does |
|---|---|
| 5-day look-back / look-ahead date window | No date window exists in `js/reconciliation.js` |
| Inflow 3-tier keyword rule (2+ keywords = strong; 1 keyword + amount within 30% = good; exact amount = fallback) | No keyword tiers in the recon engine. That keyword logic lives in `js/cashflow.js:69` for the forecast clearing rule, not here |
| Outflow costs need **all three**: keyword from cost name + amount within **33%** + compatible sub-category | No keyword gate, no sub-category gate. Real tolerance is `COST_MATCH_TOLERANCE = 0.10` (10%) at [js/reconciliation.js:498](js/reconciliation.js:498) |

**Why this one matters.** The SOP states a 33% amount tolerance where the code enforces 10%. On a
genuine supplier price rise the SOP implies the match still lands; the code deliberately returns
blank and leaves it for a human. That is a real behavioural difference on money attribution, and
it sits directly beside the three cost-matching bugs fixed yesterday (87d6b62, 78220d3, 5ed9137).
