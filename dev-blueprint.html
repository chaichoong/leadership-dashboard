1. Repo Structure & File Ownership

The monolith was split so parallel Claude sessions can build without overwriting each other. One feature, one file.

Golden rule: two sessions must never edit the same file at the same time. Shared files (js/config.js, js/shared.js, index.html, css/styles.css) are edited by ONE session, in the same session as the feature needing the change. Always git pull before starting, push promptly after committing.

2. JS Internals — Globals, Helpers, Conventions

All JS loads as plain <script> tags in one global scope. No bundler, no modules. Load order: config.js → shared.js → feature files.

🌍Global variablesjs/config.js + dashboard.js
  • PAT — Airtable auth token, set by the auth flow. Never log it, never hardcode it.
  • Data arrays loaded by dashboard.js: allTransactions, allTenancies, allTenants, allCosts, allCategories, allSubCategories, allBusinesses
  • Constants: F (field IDs), TABLES, INV, REC, PS
  • AI models: AI_MODEL_DEFAULT, AI_MODEL_LIGHT — feature files must use these, never a literal model ID. The literals live once in js/ai-models.js (loaded before config.js); pages that cannot load config.js read window.AI_MODELS.default / .light directly
Rule: prefer the shared arrays over independent fetches. Only call Airtable separately for tables not already cached.
🧰shared.js helper cataloguejs/shared.js
  • Data: airtableFetch(tableId, params) (handles pagination) · getField · getNumVal · lookupValues · txDisplayAmount · getActiveBusinesses
  • Status logic: isTenancyActive · isTenancyIncome · isTenantStatusActive/Former · isCostActive · getPaymentStatusName
  • Security: escHtml(str) for ALL external text into HTML · escJs(str)
  • UI: expandableCard · toggleCard · switchTab(tabId) · showToast · showConfirm · openQuickTaskModal
  • Health: updateSidebarHealth(tabId, status) · sync bars via registerSyncBar in js/sync-bar.js
  • Robustness: limitedApiFetch (rate limit) · nextRenderGen/isCurrentRender (render race guard) · smartRefresh + idle auto-refresh
⚠️Known anti-patterns (real production bugs)Check every build
  • Missing Number() on PATCH — Airtable number fields reject strings: fields: { [F.amount]: Number(value) }
  • Partial re-render stale state — after inline edit/status/filter change call the full renderAll(), not the single-item updater
  • returnFieldsByFieldId — this project uses field NAMES via F constants; field-ID keys silently read undefined
  • CSS overflow truncation — long names need ellipsis or word-break: break-word
  • localStorage quota — Safari caps at 5MB; large caches go to IndexedDB (dashboard.js pattern), localStorage is small UI state only, keys namespaced per feature (cfv_, recon_…)
  • Linked-record filters — filter by record ID, never ARRAYJOIN display names
🚫Forbidden patternsNever introduce
  • console.log / debugger in production paths
  • Silent catch blocks (must log or toast)
  • Secrets or PAT tokens hardcoded anywhere
  • Hardcoded AI model IDs in feature files (a retired ID is an app-wide AI outage)
  • eval(), document.write(), Function() constructor
  • innerHTML with unescaped external data — use escHtml()
  • Hardcoded colours or fonts — design tokens only (§1 of CLAUDE.md design system)

3. Deploy Pipeline

The repo is the source of truth. Push to main → GitHub Pages auto-deploys in 2-3 minutes → confirm live before declaring done.

🚀Deploy flowPipeline
  • Live URL: https://chaichoong.github.io/leadership-dashboard/
  • git push origin main → Pages build → poll the deploy monitor script in the background until live
  • Never say "done" after pushing — the task completes when the deploy is confirmed live (watch stale browser cache; hard reload)
  • Rollback = identify the revert commit BEFORE deploying anything risky
🔢Version tracking (PAGE_REGISTRY)Automation
  • PAGE_REGISTRY in js/config.js tracks pageVer / sopVer per page
  • .github/workflows/auto-bump-pagever.yml bumps pageVer automatically when a page's source file hits main
  • File→page mapping lives in scripts/pre-commit-action.py — update it when adding pages
  • Local hooks: scripts/pre-commit (version bump) and scripts/pre-push (test gate) — symlink into .git/hooks/
  • sopVer is bumped manually when the SOP catches up with the page
⏰Scheduled jobsGitHub Actions + local
  • monthly-valuations.yml — property valuations run, 1st of the month (needs PROXY_SERVICE_TOKEN secret)
  • sync-ai-brain.yml — publishes the AI Brain feed
  • sync-transactions.yml — transaction sync
  • scripts/sync-master-plan.py — nightly plan↔Airtable sync: ticks MASTER-PLAN from Airtable completions, pushes approved plan tasks to Airtable

4. Testing & Quality Gates

Nothing ships unverified. Kevin never debugs — the session that builds it proves it works.

🔒Sync-invariant suitePlaywright
  • tests/sync-invariants/ — 11 spec files (33 tests): active-business filter, auth token handling, badge count sync, cascade sync, CFV detection, no-crash-on-load, and more
  • Run: npm run test:sync
  • A local pre-push gate (not committed to the repo) blocks failing pushes to main
🧪Unit testsVitest
  • tests/shared.test.js — shared helper coverage
  • The /test-gaps skill generates Vitest tests for untested paths in changed files: data writes and business logic first, trivial getters skipped
✅Quality pipeline (every build/fix)Skills
Order, run by /build-feature and /fix automatically:
  • Self-audit → /verify (browser proof) → simplify pass → /test-gaps → independent review gate (blocking) → security review when auth/data/money → /pre-deploy checklist → deploy → confirm live → /test against production
Rule: no "done" without browser-verified proof. Screenshot evidence for UI changes.

5. Supabase Migration — the Shadow Build

Airtable stays the system of record until per-module cutover. A parallel Supabase/Vercel build runs alongside via twins and shims. Spec: docs/supabase-schema-spec.md.

👯Twin pages + fetch shimsPattern
  • supabase-app.html — the shell for the migrated app (deploys to Vercel)
  • *-supabase.html twins mirror live pages: index, accounts, pnl, money, wealth, ai-brain, follow-up, skills
  • *-shim.js (5: dashboard, wealth, ai-brain, comms, skills) override window.fetch and translate Airtable REST calls into Supabase queries — so the original feature JS runs unmodified
  • Shims load NON-deferred, before feature code, and can chain (a page can load several)
  • Shims await the Supabase session before the first query (fixes blank-data-on-load)
🗄️Migrations statesupabase-migration/
  • In repo: 0015_systemisation · 0016_app_settings · 0017_objectives_strategy · 0018_ai_brain · 0019_wealth
  • 0001–0014 applied to the project but pending commit to the repo (MASTER-PLAN §4 task, Mica)
  • Tenancy spine applied: tenants, tenant_users, tenant_config, ai_usage_log
  • Open: RLS policy audit, tenant_id retrofit on every table, entities table (D1)
🧭Cutover rulesRule
  • Per-module cutover, not big-bang. A module cuts over when its twin loads real data on Vercel and passes the parity check
  • Until then every write goes to Airtable; the shadow build is read-mirrored
  • Replace any Airtable automation a migrating page depends on before its cutover
  • Target: launch on Vercel + Supabase by early August 2026

7. AI Layer

Everything AI-facing: model routing, prompts, skills, memory. The north star: AI agents execute up to 90% of repeatable operational work.

🎛️Model routingjs/config.js
  • AI_MODEL_DEFAULT — main model for in-app AI features
  • AI_MODEL_LIGHT — cheap model for light tasks
  • Feature files import these constants. A hardcoded model ID that later retires takes down every AI feature at once — this happened; do not repeat it
  • Requests route through the claude-proxy worker; per-tenant AI keys (D9) are in build
🎙️Promptsjs/prompts/
  • js/prompts/boardroom-mentor.js — the single source of truth for every AI wizard's voice. All wizards import it; never fork the tone per feature
  • js/ai-assistant.js — the chat panel; per-tab context blocks (e.g. ctx.compliancePage) tell the AI what the user is looking at. New features that hold data Kevin might ask about add a context block here
⚡Claude Code skills.claude/skills/
In-repo skills that drive the workflow:
  • build-feature — end-to-end build pipeline (BILD → plan → build → audit → review → deploy)
  • fix — same pipeline for bugs and amendments
  • audit · verify · test · test-gaps · pre-deploy · health-bar
Kevin's global skills (transcript-to-brain, task creators, etc.) live outside the repo in ~/.claude/skills/.
